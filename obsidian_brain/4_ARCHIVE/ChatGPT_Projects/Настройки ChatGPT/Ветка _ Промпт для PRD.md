# Product Requirements Document (PRD)

# <Имя продукта — вывести из BRIEF>



## 1. Executive Summary

— 1 абзац: бизнес-контекст и «работа, которую нужно сделать».

— Ценность, ключевые актёры, критерии успеха (S.M.A.R.T.).



## 2. Область и пользовательские потоки

— Основные сценарии: шаги + «happy-path» (проза, нумерация).

— Крайние случаи: ошибки, повторы, отмены, простоевый режим.

— Для ботов/интерфейсов: команды, антиспам/троттлинг, роли.



## 3. Функциональные требования

— Перечень фич с **Acceptance Criteria (Given/When/Then)**.

— Матрица конфигурации (что настраивается и где хранится).

— Роли/доступы (Admin/Engineer/QA/Viewer/Bot/Anon).



## 4. Архитектура (bolt.new-ready при наличии ИТ-части)

— Frontend (если есть): React 18+ TS, Tailwind, lucide-react; стратегия состояния.

— Backend/Инфра: Supabase (Postgres/Auth/Realtime) + Edge Functions (Deno) ИЛИ корпоративный стек (PDM/PLM/1C/APIs).

— Интеграции: точные API/версии; схема аутентификации.

— Последовательности (проза): критические пути (бронь/отмена/синхрониз./выдача КД/извещение об изменениях).



## 5. Схема данных / Модель предметной области

— Сущности и связи (HVAC: системы/ветки/узлы/оборудование/маркировка; Оснастка: узлы/детали/операции/ТС/ТК/комплекты).

— Версионирование КД/ТП; статусы; атрибутика (обозначения, ревизии).

— Файловые форматы (DWG/DXF/STEP/PDF/DOCX), правила именования.



## 6. Security & RLS Policies (если есть БД/веб)

— RLS на таблицу: **кто может SELECT/INSERT/UPDATE/DELETE**.

— Принцип: запись **только** через Edge Functions (service-role); анонимам — read-only по белому списку.

— Хранение секретов: **Supabase Secrets/Vault** (не в таблицах).

— Проверка Telegram-webhook: заголовок X-Telegram-Bot-Api-Secret-Token.

— Валидация/санитизация входных данных. Логи попыток.



## 7. Интеграции — детали

— Точные эндпоинты, скоупы, лимиты; стратегии ретраев/фолбэков.

— CAD/PDM/PLM: обмен (DWG/STEP/PLM-карточки), правила синхронизации атрибутов.

— Почта/мессенджер: шаблоны уведомлений, SLA доставки.



## 8. Управление изменениями и трассируемость

— Жизненный цикл ревизий: статусы, согласование, протоколы.

— Автоматические уведомления, журнал причин изменения.

— Связь КД ↔ ТП ↔ изделие/узел ↔ партия/место установки.



## 9. Локаль, единицы, часовые пояса

— СИ, формат дат/времени; внутренняя UTC + преобразования.

— PRIMARY_TZ из ввода или дефолт; политика DST.



## 10. UI/UX требования (при наличии интерфейса)

— Карта навигации, инвентаризация компонентов.

— Состояния: loading/empty/error/success/skeleton.

— A11y основы (порядок фокуса, aria), брейкпойнты.



## 11. Нефункциональные требования (SLOs) + ссылки на НТД

— Надёжность/доступность/производительность (TTFB/FCP/API p95).

— Наблюдаемость: структурированные логи, трассировка, алёрты.

— Резервное копирование/восстановление; политика хранения данных/PII.

— **Нормативная база (пример, актуализацию уточнить):**

   - **ЕСКД (ГОСТ 2.\*)**, **ЕСТД (ГОСТ 3.\*)** — оформление КД/ТП, изменения, обозначения.

   - **СП 60.13330** (ОВК), **СП 73.13330** (внутренние системы), **СП 48.13330** (организация строительства) — применимо к HVAC.

   - **ТР ТС 010/2011** (безопасность машин/оборудования) — для оснастки.

   - **ISO 9001** (качество, управл. изменениями); **ISO 12100/13849** (безопасность машин) — при необходимости.

— Для каждого пункта укажи: требование → метрика/порог → метод контроля → ссылка на норму.



## 12. Стратегия тестирования + ссылки на НТД

— Unit/интеграционные/E2E (Playwright) для ИТ-части; контрактные тесты внешних API (mock).

— Приёмочные испытания (чек-лист **Given/When/Then**).

— Для оснастки/производства: входной контроль, пооперационный, приёмосдаточные, НК (ВИК), карты контроля.

— **Нормативы (пример, актуализировать):** серия **ГОСТ 34** (АС), **ЕСТД** (оформление ТК/ЭТК/Маршрутных карт), **СП/ГОСТ** по видам работ (HVAC), корпоративные СТО.



## 13. Развёртывание и конфигурация (если ИТ)

— Переменные окружения (frontend vs edge).

— Команды сборки/CI; post-deploy health-checks; план отката.



## 14. Риски и меры

— Топ-риски (аутентификация, квоты, гонки, TZ, ошибки КД/версии, поставки).

— Тактики снижения + мониторинговые хуки.



## 15. Метрики успеха

— Северная звезда + 3–5 KPI, жёстко связанные с ценностью (напр., % трассируемых изменений КД; TTM узла оснастки; срок устранения замечаний СТК/ИД; % закрытых наряд-заказов без возврата).

📝 Отчёт по выполненной работе
1. Цель работы

Попытаться собрать, очистить и подготовить 3D-деталь так, чтобы она не разваливалась на атомы при экспорте. А также — заставить Blender, Bolt и CAD-сущности хотя бы притвориться, что они дружат.

2. Выполненные действия

Импортировано CAD-изображение детали
— Blender встретил его с таким же энтузиазмом, как кот — ванну.
— Выявлено: куча паразитных тел, направляющих, обрывков геометрии и других приветиков из CAD-мультвселенной.

Проведена чистка модели
— Удалены лишние объекты.
— Нормали сведены к миру и гармонии.
— Меши объединены, поправлены и выровнены.
— Приложено много моральных усилий.

Оптимизация под экспорт
— Удалены doubles, устранены артефакты, нормали переведены в приличное состояние.
— Создан более или менее цельный меш, который хотя бы выглядит как внушающий доверие объект.

Воссоздана геометрия вручную
— Места, где CAD просто сказал “я устал” — восстановлены вручную.
— Объект доведён до состояния: “Ну, хотя бы ломаться теперь будет красиво”.

Подготовлена финальная версия для Bolt/дальнейшего использования
— Меш очищен.
— Топология приводена к цивилизации.
— Проверено на артефакты.

3. Результаты

— Модель больше не похожа на кошмар первокурсника-машиностроителя.
— Геометрия структурирована и экспортируется без того, чтобы Bolt начинал молиться тёмным богам CADа.
— Получена чистая, упрощённая и пригодная для работы версия детали.

4. Проблемы, встреченные в процессе

— CAD решил прислать не модель, а послание: “Разберись как хочешь”.
— Нормали творили акробатические номера.
— Неоднозначные элементы меша заставили Blender нервно подёргиваться.
— Приходилось местами восстанавливать геометрию вручную, потому что оригинал говорил: “Не-не-не, это уже не мои проблемы”.
